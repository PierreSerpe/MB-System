41 /*
42  * Notes on the mbsys_swathplus data structure and associated format:
43  *
44  * In early 2013 SEA sold the SWATHplus system to BathySwath where
45  * the system will be rebranded the BathySwath.
46  *
47  * 1. BathySwath defines three data formats associated with the SWATHplus
48  * interferometric sonar: raw, intermediate, and processed.
49  * MB-System supports the intermediate format as MBIO
50  * format 221 (MBF_SWPLSSXI) and the processed format as MBIO
51  * format 222 (MBF_SWPLSSXP). Only the SXP format is fully tested at this time.
52  *
53  * 2. Bathyswath is a swath bathymetry sonar system. It is derived from
54  * the SWATHplus sonar system, and uses the same file formats. In
55  * turn, SWATHplus was derived from the Submetrix sonars, built by
56  * Submetrix Ltd.
57  *
58  * 3. The data files are written using a Microsoft Windows operating system
59  * and therefore follow the conventions of that system in terms of file
60  * naming and low-level disk format. In particular, data structures
61  * often contain padding bytes that must be preserved to maintain
62  * compatibility with the format.
63  *
64  * 4. All Bathyswath and SWATHplus data is little-endian, i.e. in the
65  * natural 80x86 format with the least significant byte at the lower
66  * address.
67  *
68  * 5. MBF_SWPLSSXP - Processed data derived from the real-time software.
69  * These files have all corrections applied, including: attitude,
70  * position, tide, speed of sound, includes down-sampled position,
71  * attitude and tide information. Processed files use a projected
72  * coordinate system (PCS), typically UTM. The user must supply PRJ
73  * files for each input sxp file identifying the correct PCS.
74  *
75  * 6. MBF_SWPLSSXI - Raw data, but parsed into a format that is easier for
76  * third-party code to interpret. These files have none of the above
77  * corrections applied.
78  *
79  * 7. All of the bathyswath files use the same block-oriented data format.
80  * They can be read using the same software code, and the blocks that
81  * they contain may be included in any of the files. The difference
82  * between the file types is therefore simply the types of data block
83  * that they tend to contain. Each file contains a file header block,
84  * followed by a series of data blocks. Every block contains a header
85  * that identifies the block, followed by the length of the block.
86  * Therefore, the reading software can identify the blocks that it
87  * wishes to read and ignore and skip over any block types that it
88  * encounters. In this way, new blocks can be added to a file without
89  * necessarily having to update the reading software.
90  *
91  */
