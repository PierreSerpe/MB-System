32 /*
33  * Notes on the MBF_BCHRXUNB data format:
34  * 1. Elac multibeam systems output binary data telegrams.
35  * This format differs from the original "UNB" format in
36  * that the depth values are in 4-byte rather than 2-byte
37  * integers. This "extended" format supports deeper depths
38  * than the original format.
39  * 2. Elac BottomChart sonar systems output both bathymetry
40  * and amplitude information for up to 56 beams per telegram.
41  * 3. Each ping produces 8 beams. A wide swath is constructed
42  * by successively pinging in different directions.
43  * 4. Each telegram is preceded by a two byte start code and
44  * followed by a three byte end code consisting of 0x03
45  * followed by two bytes representing the checksum for
46  * the data bytes. MB-System does not calculate checksums
47  * and puts 0's in the checksum bytes.
48  * 5. The relevant telegram start codes, types, and sizes are:
49  * 0x0250: Comment (Defined only for MB-System) 200 data bytes
50  * 0x0251: Position 36 data bytes
51  * 0x0252: Parameter 54 data bytes
52  * 0x0253: Sound velocity profile 2016 data bytes
53  * 0x0254: BottomChart 56 beam bathymetry 848 data bytes
54  * 0x0255: BottomChart 40 beam bathymetry 608 data bytes
55  * 0x0256: BottomChart 32 beam bathymetry 488 data bytes
56  * 6. Elac systems record navigation fixes using the position
57  * telegram; navigation is not always included in the per ping data.
58  * Since speed is not recorded, it is impossible to extrapolate
59  * position from the last navigation fix when processing the
60  * data serially, as MBIO does. It may thus be necessary to extract
61  * the navigation from the position telegrams and remerge it with
62  * the ping telegrams using the program mbmerge.
63  *
64  */
