36  * MBF_EM300RAW : MBIO ID 56 - Vendor EM3000, EM300, EM120
37  * MBF_EM300MBA : MBIO ID 57 - MBARI EM3000, EM300, EM120 for processing
38  * MBSYS_SIMRAD2 formats (code in mbsys_simrad2.c and mbsys_simrad2.h):

47 /*
48  * Notes on the MBSYS_SIMRAD2 data structure:
49  * 1. Simrad multibeam systems output datagrams which are
50  * a combination of ascii and binary. This code has been written
51  * using data format specifications found in an April 28, 1998
52  * technical note from Simrad.
53  * 2. Simrad multibeam sonars output both bathymetry
54  * and amplitude information for beams and sidescan information
55  * with a higher resolution than the bathymetry and amplitude.
56  * 3. There are three systems of interest:
57  * EM-3000: Single array 300 kHz shallow water system with up to 127
58  * beams of bathymetry and a variable number of sidescan
59  * samples per bathymetry beam.
60  * EM-3000D: Double array 300 kHz shallow water system with up to 254
61  * beams of bathymetry and a variable number of sidescan
62  * samples per bathymetry beam.
63  * EM-2000: Single array 200 kHz shallow water system with up to 87
64  * beams of bathymetry and a variable number of sidescan
65  * samples per bathymetry beam.
66  * EM-1002: Single array 95 kHz shallow water system with up to 111
67  * beams of bathymetry and a variable number of sidescan
68  * samples per bathymetry beam.
69  * EM-300: Single array 30 kHz mid water system with up to 135
70  * beams of bathymetry and a variable number of sidescan
71  * samples per bathymetry beam. This system applies
72  * pitch and yaw compensation to achieve
73  * more uniform coverage of the seafloor.
74  * EM-120: Single array 12 kHz full ocean system with up to 191
75  * beams of bathymetry and a variable number of sidescan
76  * samples per bathymetry beam. This system applies
77  * pitch and yaw compensation to achieve
78  * more uniform coverage of the seafloor.
79  * 4. Each telegram is preceded by a two byte start code 0x02 and
80  * followed by a three byte end code consisting of 0x03
81  * followed by two bytes representing the checksum for
82  * the data bytes. MB-System does not check the checksums
83  * on input, but does calculate the checksums for output.
84  * 5. The Kongsberg Simrad datagram format manual lists a large number
85  * of datagram types. The complete list of telegram start codes,
86  * types, and sizes is given below. Datagram listings preceded
87  * by an "*" are recognized by MB-System. Unrecognized datagrams
88  * will be skipped on input and not included in output files.
89  * *0x0231: Parameter - Data out off variable size
90  * *0x0232: Parameter - Data out on variable size
91  * *0x0230: Parameter - Stop variable size
92  * *0x0241: Attitude Output 1222 bytes
93  * *0x0243: Clock Output 28 bytes
94  * *0x0244: Bathymetry 48-4092 bytes
95  * 0x0245: Single beam echosounder depth 32 bytes
96  * *0x0246: Raw range and beam angle 24-2056 bytes
97  * *0x0247: Surface sound speed variable size
98  * *0x0248: Heading Output 422 bytes
99  * *0x0249: Parameter - Start variable size
100  * *0x024A: Mechanical transducer tilt variable size
101  * 0x024B: Central beams echogram variable size
102  * *0x0250: Position 100-134 bytes
103  * *0x0252: Runtime Parameter 52 bytes
104  * *0x0253: Sidescan 48->5K bytes
105  * *0x0254: Tide Output 30 bytes
106  * *0x0255: Sound velocity profile (new) variable size
107  * *0x0256: Sound velocity profile (old) variable size
108  * 0x0257: SSP input variable size
109  * *0x0265: Raw range and beam angle 112-1658 bytes
110  * *0x0266: Raw range and beam angle 44-1658 bytes
111  * *0x0268: Height Output 24 bytes
112  * *0x0269: Parameter - Stop variable size
113  * *0x026B: Water column variable size
114  * 0x0270: Parameter - Remote variable size
115  * 0x0273: Surface sound speed variable size
116  * *0x02E1: Bathymetry (MBARI format 57) 48-4092 bytes
117  * *0x02E2: Sidescan (MBARI format 57) 48->5K bytes
118  * 6. Simrad systems record navigation fixes using the position
119  * datagram; no navigation is included in the per ping data. Thus,
120  * it is necessary to extrapolate the navigation for each ping
121  * at read time from the last navigation fix. The frequency of
122  * GPS fixes generally assures that this is not a problem, but
123  * we offer no guarantees that this will always be the case.
124  * In this format the navigation fix datagrams include copies of
125  * the ASCII data records (typically NMEA 0183) input into the
126  * sonar by the navigation system.
127  * 7. The beam depths are give relative to the transmit transducer
128  * or sonar head depth and the horizontal location of the active
129  * positioning system's antenna (or reference point). Heave,
130  * roll, pitch, sound speed at the transducer depth and ray
131  * bending have been applied. In the EM3000 the beam depths must
132  * be regarded as signed values to take into account beams
133  * which may be going upwards. On the EM300 the beam depths are
134  * always positive and the values are therefore unsigned.
135  * To obtain depths relative to the water line, the
136  * raw depths must be added to the transmit transducer depth plus
137  * the depth offset multiplier times 65536 cm. The depth offset
138  * multiplier will be zero except when
139  * 1) the EM3000 sonar head is on an underwater vehicle
140  * at a depth greater than 650 m, or
141  * 2) when the heave is large enough to bring the transmit
142  * transducer above the water line (the depth offset
143  * multiplier is -1 in this case).
144  * 8. The new Simrad sonars record the heading and attitude sensor
145  * data streams input into the sonar, usually at a sampling
146  * frequency of 100 Hz.
147  * 9. Although this new format started out as a fairly clean conception,
148  * later revisions adding new information have been implemented in
149  * an unnecessarily complicated fashion. For example,
150  * if the sonar is an EM300 or EM3000, then the sample rate
151  * value contains the sample rate in Hz as an unsigned short.
152  * The range resolution in time is half the inverse of the
153  * sampling rate, or approximately 2.5 cm for an EM3000.
154  * HOWEVER, if the sonar is an EM3000D, then the sample rate value
155  * contains the depth difference between the two sonar heads
156  * (the units are not specified in the spec, but are presumably in
157  * cm like the transmit transducer depth). In this case, the transmit
158  * transducer depth for the second sonar head is the transmit
159  * transducer depth plus the depth difference. The sample rate for the
160  * EM3000D is derived from the sonar model, as follows:
161  * Sonar Model Head 1 Sample Rate Head 2 Sample Rate
162  * ----------- ------------------ ------------------
163  * 3002 13956 Hz 14621 Hz
164  * 3003 13956 Hz 14621 Hz
165  * 3004 14293 Hz 14621 Hz
166  * 3005 13956 Hz 14293 Hz
167  * 3006 14621 Hz 14293 Hz
168  * 3007 14293 Hz 13956 Hz
169  * 3008 14621 Hz 13956 Hz
170  * In the case of sonar model 3002, the depth difference should be
171  * taken as zero, regardless of the value found in the data.
172  * 10. An additional complication involves the beam angles and travel
173  * times reported in the data stream. In the original format
174  * specification, the beam range is given as a one way travel time,
175  * the beam depression angle is positive downwards (90 deg at vertical)
176  * and the beam azimuth angle gives the orientation of the sounding
177  * relative to the heading of the vessel. These values include the
178  * effects of the vessels motion during the ping cycle (heave, pitch,
179  * roll, and yaw), and are sufficient for recalculating bathymetry
180  * by raytracing (an improvement over older Simrad data formats).
181  * HOWEVER, the sonar operator can specify a data recording mode in
182  * which the raw times and angles are recorded without correction
183  * for the vessel motion. In this case, the raw two-way travel times
184  * are recorded instead of one-way travel times. Also, the beam
185  * depression angles are replaced by beam pointing angles, which are
186  * the angles relative to the sonar heads before correction for roll
187  * and roll bias. The beam azimuth angle is replaced by a combination
188  * of a flag signaling the raw data mode (angle values greater than 35999
189  * indicates storage of uncorrected range and angle data) and the
190  * transmitter tilt angle (angle value minus 54000 in 0.01 deg). The
191  * purpose of the uncorrected data mode is to allow users to
192  * fully recalculate the data following revision of the heading and
193  * attitude data streams.
194  * NOTE: As of May 2002, Simrad seems to have implemented a different
195  * scheme for the EM300 and EM120 sonars. There is a new raw range and
196  * angle datagram (0x0265) that is output in addition to the depth and
197  * seafloor image datagrams. This would seem to make obsolete the notion
198  * of placing uncorrected ranges and angles in the depth datagram.
199  * 11. The sidescan is structured in terms of a certain number of samples
200  * per beam. The range sampling rate for the sidescan is the same as
201  * that specified in the depth datagram and the ranges in the sidescan
202  * datagram (seabed image datagram) are all two way travel times.
203  * 12. The attitude data is output asynchronously with respect to the
204  * ping output datagrams. Typical motion sensors give data at a 100 Hz
205  * rate. The attitude datagrams are given when the number of measurements
206  * is 100, or usually at 1 second intervals. The attitude data timing
207  * is corrected for the sensor time delay entered by the operator. If
208  * roll is input with respect to the horizontal, then the sonar
209  * recalculates the roll so that the output values are in the plane
210  * defined by the heading and pitch axis. The entered sensor offsets
211  * (roll bias, pitch bias, heading bias). Extra heave at the transducer
212  * due to roll and pitch when the sensor does not give its data at the
213  * transducer position is also included and heave is positive downwards.
214  * The sensor status is copied from the input datagram's two sync bytes
215  * with the second byte always set to 0x90. The first byte is either
216  * zero or in the 0x90-0xAF range. If the latter is true, then 0x90
217  * indicates valid data with full accuracy, 0x91-0x99 indicates valid
218  * data with gradually reduced accuracy, 0x9A-0x9F indicates invalid data
219  * from an operating sensor, and 0xA0-0xAF indicates invalid data
220  * from a faulty sensor. This interpretation may be dependent on the
221  * attitude sensor type.
222  * 13. The heading data is output asynchronously with respect to the
223  * ping output datagrams. Typical heading sensors give data at a 10 Hz
224  * rate. The attitude datagrams are given when the number of measurements
225  * is 100, or usually at 10 second intervals. The heading data is
226  * corrected for the heading offset entered by the operator.
227  * 14. The raw vendor format (format 56) does not support flagging of bathymetry
228  * values nor does it store navigation in the bathymetry data records.
229  * MB-System also supports a processing format (57) which includes
230  * beamflags and navigation in the bathymetry records. This format
231  * is identical to the vendor format except for the use of a
232  * slightly different bathymetry record.
233  *
234  */
