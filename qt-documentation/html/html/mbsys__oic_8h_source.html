27  * MBF_OICGEODA : MBIO ID 141
28  * MBF_OICMBARI : MBIO ID 142
29  *
30  * Author: D. W. Caress
31  * Date: March 1, 1999
32  *

34 /*
35  * Notes on the MBSYS_OIC data structure:
36  * 1. This data structure is used to store swath sonar data
37  * collected and recorded using the GeoDAS-SV1 system
38  * developed and sold by Ocean Imaging Consultants.
39  * In particular, the Deep Submergence Lab (DSL) of
40  * the Woods Hole Oceanographic Institution (WHOI) now
41  * uses an OIC GeoDAS-SV1 package as the front end for
42  * their deep towed 120 kHz interferometry sonar (AMS-120).
43  * 2. Two data formats are supported. The first, MBF_OICGEODA,
44  * is the vendor format containing unflaggable bathymetry
45  * and raw sidescan. The MBIO i/o module for this format
46  * generates complete bathymetry and sidescan out of the
47  * raw information, but does not support flagging of bad
48  * bathymetry or the storage of corrected sidescan.
49  * The second, MBF_OICMBARI, is intended as a processing
50  * and archiving format as it allows the storage of both
51  * the unaltered raw values and editable and correctable
52  * bathymetry and sidescan.
53  * 3. In both formats the data consist of variable length
54  * binary records.
55  * 4. Each data record has three parts. First there is a 248-byte
56  * header section containing the time stamp, navigation, a
57  * variety of other values, the numbers and sizes of sonar
58  * data included in the record. The second, optional, part of
59  * the record is "client specific information" of arbitrary
60  * length. The third part contains up to eight arrays of
61  * "raw" sonar data followed by, in the MBF_OICMBARI format,
62  * arrays of processing bathymetry and sidescan.
63  * 5. The header begins with a four byte magic number. For the
64  * MBF_OICGEODA format the first three bytes are 'G', 'E',
65  * and 'O'. For the MBF_OICMBARI format the first three bytes
66  * are 'G', 'E', and '2'.The fourth byte is the id number for
67  * the data source.
68  * 6. The maximum numbers of beams and pixels defined below are
69  * limitations specific to MBIO - the formats themselves
70  * contain no limits on the amount of data per record.
71  * 7. Comment records are encoded in MBIO using the "client
72  * specific information" section of the record and an unused
73  * sonar type value.
74  *
75  */
